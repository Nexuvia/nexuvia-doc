# Migrating 0.1.x ‚Üí 0.2.0

Version 0.2.0 introduces breaking changes to `AuthClientConfig` and new conditional exports on several packages. Follow the steps below in order.

---

## 1. Update `AuthClientConfig` in `nexuvia.config.ts`

`azure.redirectUri` has been removed from the top-level config. The redirect URI now lives inside `AzureStoreConfig` returned by `storeConfigProvider` ‚Äî this allows each store to have its own redirect URI.

**Before:**

```ts
authClient: {
  session: {
    cookieName:      'auth_session',
    nonceCookieName: 'auth_nonce',
    storeCookieName: 'auth_store',
    encryptionKey:   process.env.AUTH_ENCRYPTION_KEY || '',
    secureCookies:   true,
  },
  azure: {
    redirectUri: 'https://example.com/auth/callback',
  },
},
```

**After:**

```ts
authClient: {
  session: {
    encryptionKey: process.env.AUTH_ENCRYPTION_KEY || '',
    secureCookies: process.env.NODE_ENV === 'production',
    // cookieName, nonceCookieName, storeCookieName are now optional ‚Äî defaults apply
  },
  storeConfigProvider: async (storeKey) => getAzureStoreConfig(storeKey),
  // redirectUri now lives inside the AzureStoreConfig returned above
},
```

`storeConfigProvider` is now **required** when `authClient` is configured. It replaces the static `azure.redirectUri`.

### Update `config/auth.ts`

Your `storeConfigProvider` implementation must now return a `redirectUri` per store:

```ts
async function getAzureStoreConfig(storeKey: string): Promise<AzureStoreConfig> {
  // ...fetch authority, clientId, clientSecret from Hybris config endpoint...
  return {
    authority,
    clientId,
    clientSecret,
    scope: `openid profile offline_access ${clientId}`,
    redirectUri: process.env[`AZURE_REDIRECT_URI_${storeKey.toUpperCase()}`] || '',
    // Previously this came from config.authClient.azure.redirectUri
  };
}
```

## 2. Add `nexuvia.app.ts` and adopt `NexuviaApp` (optional but recommended)

The six manual config bridge files can be replaced with a single file. This is opt-in ‚Äî the old approach still works.

**Add `nexuvia.app.ts` to your project root:**

```ts
import { NexuviaApp } from '@nexuvia/app';
import config from './nexuvia.config';

export const app = new NexuviaApp(config);
```

**Replace manual wiring in layouts:**

```ts
// Before
import { createServerOccClient, createCmsClient } from '@/config/server';

const occClient = await createServerOccClient(storeKey, lang);
const cmsService = createCmsClient(occClient);
const page = await cmsService.getContentPage('homepage');

// After
import { app } from '@/nexuvia.app';

const ctx = await app.forRequest(storeKey, lang, request);
const page = await ctx.cms.getContentPage('homepage');
```

## 3. Update subpath imports for `@nexuvia/cms` and `@nexuvia/cart`

Both packages now export `./server` and `./client` subpaths to enforce the server/client bundle boundary. The top-level import still works but the subpaths are preferred.

```ts
// @nexuvia/cms ‚Äî Before
import { CmsClient, componentRegistry } from '@nexuvia/cms';

// @nexuvia/cms ‚Äî After
import { CmsClient, OccCmsAdapter } from '@nexuvia/cms/server';  // Server Components
import { componentRegistry }         from '@nexuvia/cms/client';  // Client Components
```

```ts
// @nexuvia/cart ‚Äî Before
import { OccCartAdapter, CartClient } from '@nexuvia/cart';

// @nexuvia/cart ‚Äî After
import { OccCartAdapter }  from '@nexuvia/cart/server';  // route handlers
import { CartClient }      from '@nexuvia/cart/client';  // browser providers
```

---

## 4. Node condition on `@nexuvia/occ`, `@nexuvia/auth-client`, `@nexuvia/auth-server`

These packages now set a `"node"` export condition. Browser bundlers that don't set the `node` condition will receive the `"default"` fallback (same JS). **No code changes required** ‚Äî this is a signal to bundlers that these packages are server-only. If you were importing them in a client bundle you will now get a build warning; move those imports to server-only files.

---

## 5. Add optional cache TTL config

New `cache` field on `NexuviaConfig` ‚Äî entirely optional:

```ts
// nexuvia.config.ts
cache: {
  cmsTtl:     10 * 60 * 1000,  // override 5 min default
  productTtl:  5 * 60 * 1000,
  searchTtl:   2 * 60 * 1000,
},
```

---

## 6. Run `nexuvia check` after migrating

```bash
npx nexuvia check
```

Prints a per-field pass/fail report. All fields must be green before deploying. Common failures after this migration:

| Error | Fix |
|-------|-----|
| `authClient.storeConfigProvider ‚Äî Required` | Add `storeConfigProvider` function to `authClient` in config |
| `authClient.azure.redirectUri` validation removed | No action needed ‚Äî that field no longer exists |
| `CONFIG_INVALID: Unknown store key` | Check `storeKey` passed to `forRequest()` matches a key in `config.stores` |
